# Talep ID (No) Sistemi — Teknik Belge (23.07.2026)

**Karar sahibi:** İskender.
**Amaç:** Her talebin (ve ondan doğan tüm üretim/yayın kayıtlarının) **firma-önekli, benzersiz, insan-okur bir ID'si** olsun ve bu ID **tüm takip ekranlarında** görünsün. Böylece bir kalem ID'siyle tanınır; ileride **tekil silme** bu ID ile hedeflenir ve firma karışıklığı kaynaklı hatalı işlem önlenir.

Bu belge iki parçayı kapsar: **A — DB numaralandırma** (canlıda çalıştırılan SQL) ve **B — gösterimin tüm takibe yayılması** (kod). DB değişiklikleri repo'da başka yerde tanımlı olmadığından tüm SQL burada birebir tutulur.

---

## 0. ID mantığı (şema)

- **Formül:** `talep_no = firma_no × 10000 + firma_içi_sıra` (sıra `1..9999`).
- **firma_no:** firmanın **oluşturulma sırası** (created_at) — kalıcı, benzersiz.
- **Görünen ID:** `FirmaAdı_talep_no` (ör. `hepifarma_30001`). Metin kayıtta tutulmaz; firma adı + numaradan ekranda üretilir.
- **Kapasite:** her firmaya **9999** talep slotu. Sıra hep 4 hane; firma öneki hane sayısını büyütür:
  - firma 1 → `10001..19999`, firma 2 → `20001..29999`, … firma 9 → `90001..99999`
  - firma 10 → `100001…`, firma 11 → `110001…` ("11001 değil 110001" kuralı: sıra 4 hane kalır, önek büyür)
- **Neden firma-önekli (İskender kararı):** firma ve talep sayısı arttıkça, önek olmadan numaralar çakışır. Önek sayesinde numaranın kendisi hangi firma olduğunu kodlar; araştırırken/silerken ID karışıklığı ve hatalı işlem önlenir.

### 0.1 Önceki durumdan fark
21.07 kararında `talep_no` **firma bazlı 10001'den** başlıyordu ama **her firma 10001'den** başlıyordu (`UNIQUE(firma_id, talep_no)`; numara yalnız firma içinde benzersiz, ekranda firma adıyla ayrışıyordu). Bu belgeyle numaraya **firma öneki** bindirildi → numara **global** benzersiz oldu. Kendi 21.07 kararının bilinçli güncellenmesidir.

### 0.2 Tasarım kararı — geri sarmayan sayaç
Tekil silme geliyor → talepler silinecek. Numara "en yüksek+1" ile üretilseydi, silinen numara yeni talebe **yeniden kullanılabilirdi** (istenmeyen karışıklık). Bu yüzden firma başına **geri sarmayan bir sayaç** (`firmalar.son_talep_sira`) tutulur; silinen numara **tekrar kullanılmaz**.

### 0.3 firma_no atamaları (backfill anındaki canlı durum)
| firma_adi | firma_no | created_at |
|---|---|---|
| HapBilgi | 1 | 2026-03-27 |
| mill | 2 | 2026-03-30 |
| hepifarma | 3 | 2026-07-17 |

Talepler yalnız **hepifarma**'da vardı (7 adet, eski `10001..10007`) → yeni `30001..30007`.

## A. DB Numaralandırma (canlıda çalıştırılan SQL, tek tek)

Tüm komutlar İskender tarafından, sırayla, teker teker çalıştırıldı. Göç boyunca **talep açma yapılmadı** (eski/yeni trigger geçişinde numara çakışmasını önlemek için).

### A-1 — Okuma-teyit (salt-okuma)
Firmalar oluşturma sırası + `talep_no` aralıkları + mevcut trigger gövdesi okundu (aşağıdaki backfill varsayımı doğrulandı: hepifarma min=10001, tek bant).

Eski trigger gövdesi (`talep_no_ata`) — her firma 10001'den, MAX+1:
```sql
CREATE OR REPLACE FUNCTION public.talep_no_ata()
RETURNS trigger LANGUAGE plpgsql AS $function$
BEGIN
  IF NEW.talep_no IS NULL THEN
    SELECT COALESCE(MAX(talep_no), 10000) + 1 INTO NEW.talep_no
    FROM talepler WHERE firma_id = NEW.firma_id;
  END IF;
  RETURN NEW;
END;
$function$;
```
Trigger bağı: `talep_no_trigger BEFORE INSERT ON talepler FOR EACH ROW EXECUTE FUNCTION talep_no_ata()` (bağ değişmedi, yalnız fonksiyon gövdesi değişti).

### A-2 — Kolonlar
```sql
ALTER TABLE firmalar
  ADD COLUMN firma_no integer,
  ADD COLUMN son_talep_sira integer NOT NULL DEFAULT 0;
```

### A-3 — firma_no backfill (oluşturma sırası)
```sql
UPDATE firmalar f
SET firma_no = sub.rn
FROM (SELECT firma_id, ROW_NUMBER() OVER (ORDER BY created_at) AS rn FROM firmalar) sub
WHERE f.firma_id = sub.firma_id;
```
Sonuç: HapBilgi=1, mill=2, hepifarma=3.

### A-4 — talep_no backfill (yeni formül)
```sql
UPDATE talepler t
SET talep_no = f.firma_no * 10000 + (t.talep_no - 10000)
FROM firmalar f
WHERE t.firma_id = f.firma_id;
```
`(eski_talep_no − 10000)` = eski firma-içi sıra. firma 1 değişmez; hepifarma `10001..10007 → 30001..30007`. `UNIQUE(firma_id, talep_no)` korunur (aynı firma içinde hepsi aynı ofsetle kayar).

### A-5 — son_talep_sira backfill (sayacı kur)
```sql
UPDATE firmalar f
SET son_talep_sira = COALESCE(
  (SELECT MAX(t.talep_no) - f.firma_no * 10000 FROM talepler t WHERE t.firma_id = f.firma_id),
  0
);
```
Sonuç: hepifarma → 7, HapBilgi/mill → 0.

### A-6 — Trigger yeniden yaz (yeni şema + sayaç + 9999 koruması)
```sql
CREATE OR REPLACE FUNCTION public.talep_no_ata()
RETURNS trigger LANGUAGE plpgsql AS $function$
DECLARE
  v_firma_no  integer;
  v_yeni_sira integer;
BEGIN
  IF NEW.talep_no IS NULL THEN
    SELECT firma_no, son_talep_sira + 1
      INTO v_firma_no, v_yeni_sira
    FROM firmalar
    WHERE firma_id = NEW.firma_id
    FOR UPDATE;                       -- eşzamanlı insert'te sayaç yarışını kilitle

    IF v_firma_no IS NULL THEN
      RAISE EXCEPTION 'Firma no atanmamis (firma_id=%)', NEW.firma_id;
    END IF;

    IF v_yeni_sira > 9999 THEN
      RAISE EXCEPTION 'Firma talep kotasi doldu (firma_no=%, sinir 9999)', v_firma_no;
    END IF;

    NEW.talep_no := v_firma_no * 10000 + v_yeni_sira;

    UPDATE firmalar SET son_talep_sira = v_yeni_sira WHERE firma_id = NEW.firma_id;
  END IF;
  RETURN NEW;
END;
$function$;
```

### A-7 — Yeni firma için firma_no otomatiği
```sql
CREATE OR REPLACE FUNCTION public.firma_no_ata()
RETURNS trigger LANGUAGE plpgsql AS $function$
BEGIN
  IF NEW.firma_no IS NULL THEN
    SELECT COALESCE(MAX(firma_no), 0) + 1 INTO NEW.firma_no FROM firmalar;
  END IF;
  RETURN NEW;
END;
$function$;

CREATE TRIGGER firma_no_trigger
BEFORE INSERT ON public.firmalar
FOR EACH ROW EXECUTE FUNCTION firma_no_ata();
```

### A-8 — Kısıtlar
```sql
ALTER TABLE firmalar ALTER COLUMN firma_no SET NOT NULL;
ALTER TABLE firmalar ADD CONSTRAINT firmalar_firma_no_key UNIQUE (firma_no);
```

## A′. v_yayin_detay view'ına ID kolonları

**Neden view (kod değil):** Yayın Yönetimi ekranındaki "Yayınlar" listesi `v_yayin_detay` view'ından gelir ve view'da `talep_no`/`firma_adi`/`talep_id` yoktu. İki seçenek vardı: (a) view'a kolon ekle, (b) kod tarafı ikincil zincir sorgu.

Karar **(a) view**. Gerekçe:
- **HapBilgi dokusu:** `v_yayin_detay`'ın varlık sebebi zaten yayın zincirini düz görüntü kolonlarına indirmek (`urun_adi`, `teknik_adi`, `hedef_rol` orada denormalize). Kod yorumu: *"v_yayin_detay ile tek sorguda (eski 5 SELECT zinciri yerine)"*. `talep_no`/`firma_adi` aynı sınıf alandır; view **zaten `talepler` satırını** join'liyor.
- **4-değer:** ideal (mimari tutarlı), kaliteli (tek kaynak), sürdürülebilir (gelecek ekranlar bedava alır), verimli (0 ekstra sorgu vs 2).
- **Nüans:** Adım 3'te (Yayındaki Videolar) favori/beğeni + üreten için "kod" seçilmişti — ama onlar COUNT agregatı/kişi-join'i (hot view'ı şişirir). `talep_no`/`firma_adi` agregat değil; view zaten talep grain'inde → precedent buraya uymaz.

View değişimi (kolonlar **sona** eklendi ki `CREATE OR REPLACE` geçerli olsun; `GROUP BY` var → yeni kolonlar oraya da eklendi; `firmalar` LEFT JOIN'i eklendi):
```sql
CREATE OR REPLACE VIEW v_yayin_detay AS
 SELECT ym.yayin_id, ym.soru_seti_durum_id, ym.durum, ym.yayin_tarihi, ym.durdurma_tarihi,
    u.urun_adi, tek.teknik_adi, t.takim_id, t.uretici_id,
    t.video_basi_soru_sayisi, t.soru_seti_buyuklugu, v.video_url, v.thumbnail_url,
    vp.video_puani, avg(ssp.soru_puani)::integer AS soru_puani, ss.sorular,
    s.senaryo_metni, s.senaryo_id, sd.senaryo_durum_id, vd.video_durum_id, ssd.soru_seti_id,
    t.icerik_turu, t.hedef_rol,
    t.talep_no, f.firma_adi          -- YENİ
   FROM yayin_yonetimi ym
     JOIN soru_seti_durumu ssd ON ssd.soru_seti_durum_id = ym.soru_seti_durum_id
     JOIN soru_setleri ss ON ss.soru_seti_id = ssd.soru_seti_id
     JOIN video_durumu vd ON vd.video_durum_id = ss.video_durum_id
     JOIN videolar v ON v.video_id = vd.video_id
     JOIN talepler t ON t.talep_id = v.talep_id
     LEFT JOIN senaryo_durumu sd ON sd.senaryo_durum_id = v.senaryo_durum_id
     LEFT JOIN senaryolar s ON s.senaryo_id = sd.senaryo_id
     LEFT JOIN urunler u ON u.urun_id = t.urun_id
     LEFT JOIN teknikler tek ON tek.teknik_id = t.teknik_id
     LEFT JOIN video_puanlari vp ON vp.video_durum_id = vd.video_durum_id
     LEFT JOIN soru_seti_puanlari ssp ON ssp.soru_seti_durum_id = ym.soru_seti_durum_id
     LEFT JOIN firmalar f ON f.firma_id = t.firma_id     -- YENİ
  GROUP BY ym.yayin_id, ym.soru_seti_durum_id, ym.durum, ym.yayin_tarihi, ym.durdurma_tarihi,
    u.urun_adi, tek.teknik_adi, t.takim_id, t.uretici_id, t.video_basi_soru_sayisi,
    t.soru_seti_buyuklugu, v.video_url, v.thumbnail_url, vp.video_puani, ss.sorular,
    s.senaryo_metni, s.senaryo_id, sd.senaryo_durum_id, vd.video_durum_id, ssd.soru_seti_id,
    t.icerik_turu, t.hedef_rol,
    t.talep_no, f.firma_adi;          -- YENİ
```
Atomik, kesintisiz; satır sayısı değişmez (talep grain'i 1:1). Sonrasında şema anlık görüntüsü yenilendi: `npm run denetim:sema` → `sema.json` (v_yayin_detay 23→25 kolon; ayrıca firmalar `firma_no`/`son_talep_sira` ve `firma_no_ata` RPC yansıdı).

## B. Gösterim — ID'yi tüm takibe yay (kod)

### B.0 Ortak yardımcı (tek kaynak)
[`lib/utils/talepId.ts`](../lib/utils/talepId.ts):
```ts
export function talepIdGoster(firmaAdi?: string | null, talepNo?: number | null): string {
  if (!firmaAdi || talepNo == null) return "-";
  return `${firmaAdi}_${talepNo}`;
}
```
Tüm ekranlar bunu kullanır. Yerleşim standardı: tablolarda **ID ilk sütun** (sol kenar `px-5`, sonraki `px-3`'e kayar); kartlarda **başlıkta küçük gri satır**.
